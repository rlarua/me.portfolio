## 🔍 Trial & Error: 3일간의 학습 곡선

### ⚠️ Issue 1: 아키텍처 부재로 인한 기술 부채

#### 📍 상황 (Day 2-3)

- Vibe Coding 초기 단계에서 "기획 문서 + 디자인 가이드"만 정의
- 아키텍처 설계 없이 2개 사이트를 빠르게 구현

#### 🐛 문제

- 단일 파일 구조로 생성된 컴포넌트 (재사용성 0%)
- Props drilling 및 상태 관리 미흡
- Flat한 라우팅 구조 (스케일링 어려움)

#### 💥 임팩트

- 리팩토링 예상 공수: **추가 40시간** (전체 개발의 약 60%)
- 향후 기능 추가 시 유지보수 비용 증가

#### 💡 학습
>
> **"AI는 의도를 실행할 뿐, 설계는 엔지니어의 몫"**  
> 빠른 구현보다 **명확한 모듈화 가이드**가 장기적으로 더 효율적

---

### ⚠️ Issue 2: 컨텍스트 관리 실패

#### 📍 상황 (Day 3)

- Antigravity와 대화형으로 수정 요청을 반복
- 재시도 횟수 급증: **초기 평균 15회/기능**

#### 🐛 문제

- 의도하지 않은 코드 수정 발생 (이전 요청과 충돌)
- 컨텍스트 누적으로 Antigravity가 잘못된 방향으로 수정
- 디버깅 시간 증가

#### 🔧 해결 프로세스

```
❌ Before: 아이디어 → 바로 Antigravity 입력 (다이렉트)
             ↓
           재시도 15회, 의도 불일치

✅ After:  아이디어 → Gemini/Claude로 프롬프트 정제
             ↓
           Antigravity에 완성된 요구사항 전달
             ↓
           재시도 3회, 정확도 ↑
```

#### 📈 정량 개선

- 재시도 횟수: 15회 → **3회 (80% 감소)**
- 구현 시간: 기능당 2시간 → **40분 (67% 단축)**

#### 💡 학습
>
> **"사용량 제한 없는 도구(Gemini/ChatGPT)로 사고 정리 → 실행 도구(Antigravity)로 구현"**  
> 컨텍스트 오염을 방지하고 의도를 명확히 전달하는 2-Step 전략

---

### ⚠️ Issue 3: 품질 가이드 부재

#### 📍 발견 시점

Day 3 중후반 (2개 사이트 완성 후 코드 리뷰 시)

#### 🐛 문제

- 일관성 없는 네이밍 컨벤션
- 접근성(a11y) 미고려
- 반응형 브레이크포인트 중복 정의

#### 🔍 원인

- "빠르게 만들기"에만 집중, 품질 기준 미정의
- Antigravity는 **명시적 요구사항만 반영** (암묵적 기대는 무시)

#### 💡 학습
>
> **"AI에게 '좋은 코드'는 존재하지 않는다. 명확한 가이드만 존재한다."**

## 🎯 핵심 인사이트

### 1️⃣ AI는 "빠른 실행자"이지 "현명한 설계자"가 아니다

- 30분의 아키텍처 설계가 40시간의 리팩토링을 방지
- 개발자의 역할 변화: 코드 작성자 → **아키텍트 & 오케스트레이터**

### 2️⃣ 컨텍스트는 자산이자 부채

- 명확한 1회 요청 > 모호한 15회 대화
- 도구를 조합하여 사고를 정리한 후 실행

### 3️⃣ 명시적 가이드가 AI의 품질을 결정

- ❌ "잘 만들어줘"
- ✅ "Atomic Design 패턴 + WCAG 2.1 AA 준수"
- AI 도구에게 기대를 명확히 전달하는 것이 핵심

### 4️⃣ 학습 곡선의 압축

- 3일간의 집중 학습으로 새로운 개발 패러다임 습득
- 전통적 방식으로는 불가능한 속도로 멀티 프로젝트 완수
